You need to set up a Supabase project and configure the database before you can sync your Obsidian vault.

<Steps>
  <Step title="Create a Supabase project">
    Go to [supabase.com](https://supabase.com) and create a new project. Choose a project name, database password, and region.
    
    <Note>
      Save your database password securely - you'll need it if you want to connect to your database directly.
    </Note>
  </Step>

  <Step title="Run the database migration">
    Navigate to **SQL Editor** in your Supabase dashboard and run the following migration to create the `notes` table:

    ```sql
    -- Create notes table
    create table notes (
      id uuid primary key default gen_random_uuid(),
      user_id uuid not null,
      path text not null,
      title text,
      content text not null,
      hash text not null,
      updated_at timestamptz default now(),
      constraint notes_user_path_unique unique (user_id, path)
    );

    -- Enable Row Level Security
    alter table notes enable row level security;

    -- Revoke public access
    revoke all on table notes from anon;
    revoke all on table notes from public;

    -- Grant access to authenticated users only
    grant select, insert, update, delete on notes to authenticated;

    -- Create policies for authenticated users
    create policy "User can read own notes"
    on notes
    for select
    using (
      auth.uid() = user_id
    );

    create policy "User can insert own notes"
    on notes
    for insert
    with check (
      auth.uid() = user_id
    );

    create policy "User can update own notes"
    on notes
    for update
    using (
      auth.uid() = user_id
    )
    with check (
      auth.uid() = user_id
    );

    create policy "User can delete own notes"
    on notes
    for delete
    using (
      auth.uid() = user_id
    );

    -- Create trigger to auto-assign user_id
    create function set_user_id()
    returns trigger
    language plpgsql
    as $$
    begin
      new.user_id := auth.uid();
      return new;
    end;
    $$;

    create trigger set_user_id_trigger
    before insert on notes
    for each row
    execute function set_user_id();

    -- Create index for performance
    create index notes_user_updated_idx
    on notes (user_id, updated_at desc);

    -- Optional: Add content size limit (2MB)
    alter table notes
    add constraint content_length_limit
    check (length(content) < 2000000);
    ```

    <Warning>
      The migration must be run before you can sync notes. The table uses Row Level Security (RLS) to ensure each user can only access their own notes.
    </Warning>
  </Step>

  <Step title="Get your API credentials">
    Go to **Settings > API** in your Supabase dashboard and copy:
    
    - **Project URL**: Your Supabase project URL (e.g., `https://xxxxx.supabase.co`)
    - **Anon/Public Key**: The `anon` key (starts with `eyJ...`)

    You'll need these credentials to configure the SupaSync plugin in Obsidian.
    
    <Note>
      Use the `anon` key, not the `service_role` key. The `anon` key is safe to use in the plugin because all database operations are protected by RLS policies.
    </Note>
  </Step>
</Steps>

## Understanding Row Level Security

The `notes` table uses Supabase's Row Level Security (RLS) to ensure data isolation:

- **User isolation**: Each user can only access notes where `user_id` matches their authenticated user ID (`auth.uid()`)
- **No anonymous access**: The `anon` role is revoked - all operations require authentication
- **Automatic user assignment**: The `set_user_id()` trigger automatically sets `user_id` on insert, preventing user spoofing
- **Conflict handling**: The unique constraint on `(user_id, path)` allows safe upserts and prevents duplicate notes
